### Run profile

Per tier: 30s ramp-up + 120s hold + 30s ramp-down (3 minutes of test traffic per tier). 30s cooldown between tiers so circuit breakers, freq-cap counters, and the Redis client pool drain to a clean state before the next tier starts.

**Tiers measured:** 5,000 RPS and 10,000 RPS targets.

**15K+ RPS tiers are intentionally outside the v0 baseline scope.** The v0 baseline goal is "establish nominal + stretch reference numbers before optimisation work begins", not "find the ceiling". Stress tiers (15K, 20K, 25K, 50K) are tracked separately for a dedicated stress-tier phase where they can be compared against the Java sibling's 15K results on the same hardware.

## Interpretation

(see `docs/development-phases/phase-6.5-load-test-baseline.md` for full observation + RCA — this section is the short version)

**What's healthy:**
- HTTP latency is well under the 50 ms p99 SLA at both tiers. p95 grows from 1.4 ms at 5K to 6 ms at 10K — non-linear, but still in budget.
- Bid rate is **stable at ~70% across tiers** (5K: 70.17%, 10K: 70.18%). This is the headline correctness signal: under load, the bidder does NOT degrade its bid-fill rate as RPS climbs (after the seed-budget inflation fix; see methodology). Earlier dirty-baseline runs showed bid rate degrading from 19% → 11% across tiers — that was a budget-exhaustion artifact of compressed-time load testing, not a bidder defect.
- Resilience layer behaves as designed: breaker opens on slow-call ratio (1 open at 5K, 4 at 10K), freq-cap skip path keeps bid path moving, no cascading failure.
- `NoOpEventPublisher` and OTel-disabled paths produce zero retry noise — `kafka_events_dropped: 0` confirms the empty-broker opt-out is wired correctly.
- Per-stage timing: every stage except `frequency_cap` stays within its declared budget at 10K RPS p99.

**What's hot:**
- **`frequency_cap` p99 = 6.3 ms at 10K, max 25 ms** — the only stage materially over budget (declared 8 ms, but the budget includes the in-stage Redis MGET + decode round-trip). Under load the breaker trips because the slow-call ratio crosses threshold; freq-cap gets skipped on 552K of 1.5M requests at 10K (37%). `InProcessFrequencyCapper` is the right Phase 7 priority — eliminates this Redis hop entirely for the steady-state bid path.
- **`candidate_retrieval` budget exceeded 5,733 times at 10K (~0.4% of requests)** — RoaringBitmap intersection is occasionally slow for users with 200+ segments. Pre-computing popular-segment unions or capping per-request segment count are Phase 7 candidates.
- **Sustained RPS = 83% of target at both tiers** — k6 ramping-arrival-rate consistently delivers 4167/8333 vs 5000/10000 targets. This is k6's expected behavior on macOS Docker Desktop (file-descriptor + VU-pool ceiling), not a bidder limitation. Linux production hardware will deliver closer to 100% of target.

**What's noise:**
- 4 errors at 5K and 0 at 10K is statistical floor noise; both well below the 0.1% k6 threshold.
- The 84 ms max at 5K is the slowest single request observed, consistent with one breaker-trip cycle. Not a sustained pattern.

**Methodology constraints surfaced:**
- macOS TCP retransmit window is ~15-17 minutes vs ~5 min on Linux. Without `timeout: '5s'` on the http.post in `k6/golden.js`, a single stalled request can lock a k6 VU for the full retransmit window and inflate `max_latency` to nonsense values. Phase 6.5 added the timeout.
- The bidder's Prometheus counters are cumulative across the process lifetime. Tier N's snapshot includes tiers 1..N-1. `make baseline-tiered` now captures a `*-prometheus-before.txt` snapshot before each tier and the analyzer subtracts before-from-after to produce per-tier deltas. Without this, the Bid-rate column would show running totals.

## What this baseline does NOT measure

- Linux production hardware. macOS Apple Silicon + Docker Desktop is dev. Re-baseline on Linux x86_64 at the start of Phase 7.
- Real Kafka. Producer is `NoOpEventPublisher` for these numbers; production runs with rdkafka + a broker will differ in CPU usage but not in bid-path latency (the bid path is fire-and-forget on Kafka publish).
- Sustained 30+ minute runs. 120s hold per tier covers steady-state but not slow leaks (RSS slope, Arc cycle leaks). Phase 7 nightly chaos suite covers that.
- True production traffic shape. k6's uniform-random-with-Zipf-on-users is synthetic. Real bid traffic has time-of-day skew, campaign-spend-curve correlation with bid floor, segment-overlap clustering. Acceptable for a regression baseline; not for absolute capacity claims.

## How to re-run

```bash
# Stack + seed (one-time per fresh checkout, then idempotent)
make install-ort                     # vendors libonnxruntime
make stack-up                        # postgres + redis under bidder-stack
make migrate
make seed-postgres CAMPAIGNS=5000 SEGMENTS=1000
make seed-redis    USERS=100000 SEGMENTS=1000

# Bidder (foreground default; pass BG=1 for background daemon)
make bidder-start

# Tiered baseline (~7 min for 5K + 10K tiers; auto-restarts the bidder per tier)
make baseline-tiered

# View results
cat load-test/results/v0-baseline-summary.md
```

A diff vs this committed file shows whether a future change is a regression or improvement.
